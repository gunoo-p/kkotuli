# 꼬투리(KKOTULI) 기획서

## 1. 서비스 개요

### 1-1. 서비스 소개

끄투 온라인을 오마쥬한 웹 기반 실시간 한국어 끝말잇기 멀티플레이어 게임 플랫폼이다.
로그인 없이 방 코드 하나로 친구를 초대해 즉시 플레이할 수 있는 비회원 실시간 게임 서비스다.

### 1-2. 기획 의도

**문제 인식**

기존 온라인 끝말잇기 서비스는 다음과 같은 진입 장벽이 존재한다.

- 회원가입·로그인 절차가 필요해 즉흥적인 플레이가 어렵다
- 모르는 사람과 매칭되는 구조라 친구들끼리 즐기기 불편하다
- 모바일 환경에 최적화되지 않은 UI로 접근성이 낮다

**해결 방향**

- 방 코드 하나로 회원가입 없이 즉시 참여 가능한 구조
- 친구에게 코드를 공유하면 바로 같은 방에서 플레이
- 픽셀 아트 캐릭터 커스터마이징으로 재미 요소 추가

### 1-3. 서비스 목표

| 구분 | 내용 |
|------|------|
| 핵심 가치 | 빠르고 간단하게, 친구와 함께 |
| 타겟 사용자 | 10~30대, 친구 그룹 단위 캐주얼 게이머 |
| 핵심 기능 | 실시간 끝말잇기, 방 코드 초대, 캐릭터 커스터마이징 |
| 플랫폼 | 웹 (PC + 모바일 반응형) |

## 2. 타겟 사용자 및 사용 시나리오

### 2-1. 타겟 사용자

**주요 사용자**: 친구 그룹과 캐주얼하게 게임을 즐기고 싶은 10~30대

**사용자 특성**

- 복잡한 회원가입 절차 없이 즉시 시작하고 싶은 사용자
- 카카오톡·디스코드 등 메신저로 링크·코드를 공유하는 환경에 익숙한 사용자
- 짧은 시간(10~30분) 안에 가볍게 즐기고 싶은 사용자

### 2-2. 핵심 사용 시나리오

**시나리오 1 — 방 생성 후 친구 초대**

```
1. 사용자 A가 서비스에 접속한다
2. 닉네임과 방 옵션(두음법칙, 제한시간)을 설정하고 방을 생성한다
3. 6자리 방 코드(예: A3F9K2)를 받아 친구들에게 공유한다
4. 친구들이 코드를 입력해 대기실에 입장한다
5. 방장이 게임 시작 버튼을 누른다
```

**시나리오 2 — 게임 진행**

```
1. 게임이 시작되면 첫 단어가 제시된다
2. 현재 차례인 플레이어가 30초 안에 끝말을 잇는 단어를 입력한다
3. 서버가 끝글자·중복·사전 검증을 수행한다
4. 검증 통과 시 다음 플레이어로 턴이 넘어간다
5. 시간 초과 또는 잘못된 단어 입력 시 해당 플레이어가 탈락한다
6. 최후 1인이 남으면 게임이 종료되고 결과창이 표시된다
```

**시나리오 3 — 캐릭터 커스터마이징**

```
1. 방 참여 시 랜덤 닉네임과 픽셀 아트 캐릭터(표정·색상)가 자동 배정된다
2. 게임 시작 전 대기실에서 닉네임을 직접 수정하거나 표정·색상을 변경할 수 있다
3. 변경 사항은 같은 방 전원에게 실시간으로 반영된다
```

## 4. 화면 흐름

```
[접속]
메인 화면
    ├── 방 만들기 버튼
    │       ↓
    │   닉네임·인원·두음법칙 설정
    │       ↓
    │   방 코드 발급 → 대기실 입장
    │
    └── 방 참여 버튼
            ↓
        방 코드 입력
            ↓
        닉네임 입력 (또는 랜덤 배정)
            ↓
        대기실 입장

[대기실]
    ├── 참가자 목록 (실시간 업데이트)
    ├── 캐릭터 커스터마이징 (닉네임·표정·색상)
    ├── 실시간 채팅
    └── 게임 시작 버튼 (방장만)
            ↓
        게임 화면

[게임 화면]
    ├── 현재 단어 표시
    ├── 턴 타이머 (30초)
    ├── 참가자 목록 (생존자 표시)
    ├── 단어 입력창 (현재 턴 플레이어만 활성화)
    ├── 실시간 채팅
    └── 탈락 시 → 관전 모드로 전환
            ↓
        [게임 종료]
        결과창 (순위·사용 단어·총 턴 수)
            ↓
        다시하기 (대기실) / 나가기 (메인)

[예외 상황]
    ├── 호스트 퇴장 → 전원 메인 화면으로 이동
    └── 네트워크 끊김 → 재연결 시도 → 실패 시 메인으로 이동
```
## 5. 전체 게임 흐름

```
[방 생성]   POST /api/rooms
            → 방 코드(A3F9K2) + 세션 ID 발급 → Redis 초기화 (TTL 2시간)

[방 참여]   POST /api/rooms/join
            → 세션 ID 발급 → Redis 플레이어 등록

[WS 연결]   ws://{host}/ws?roomCode=A3F9K2&sessionId=uuid
            → /topic/room/A3F9K2 구독

[게임 시작] 방장 → SEND /app/game/start
            → 턴 순서 결정 → BROADCAST: GAME_START

[게임 진행] 턴 플레이어 → SEND /app/game/word { "word": "사과" }
            → 끝말 검증 → 사전 검증
            → 성공: BROADCAST WORD_RESULT → 다음 턴
            → 실패/타임아웃: BROADCAST TURN_TIMEOUT → 탈락

[게임 종료] 최후 1인 → BROADCAST GAME_OVER → Redis 삭제

[호스트 퇴장] SessionDisconnectEvent → BROADCAST ROOM_CLOSED → Redis 삭제
```

## 6. 협업 도구 및 규칙

### 6-1. 협업 도구

| 도구 | 용도 |
|------|------|
| Jira | 이슈 관리, 스프린트 계획 |
| GitHub | 소스 코드 버전 관리, PR 기반 코드 리뷰 |
| GitHub Actions | CI/CD 자동화 (테스트 실패 시 머지 차단) |
| Discord / Kakao Talk | 실시간 팀 커뮤니케이션 |

### 6-2. 브랜치 전략

```
main       ← 배포 브랜치 (직접 push 금지)
dev        ← 개발 통합 브랜치
feat/      ← 기능별 개발 브랜치 (예: feat/word-validation)
```

- 모든 기능 개발은 `feat/` 브랜치에서 진행하고 `dev`로 PR을 올린다.
- `dev` → `main` 머지 시 GitHub Actions CI/CD가 자동으로 배포를 수행한다.
- `main` 브랜치에 직접 push는 금지하며, 반드시 PR + 코드 리뷰를 거쳐야 한다.
